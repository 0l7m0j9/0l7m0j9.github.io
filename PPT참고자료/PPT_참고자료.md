## 2p. 기획 배경 및 아이디어 ①

**화면 내용**: 기존 유사 서비스 사례 — Spotify, 유튜브뮤직, Apple Music, FLO 아이콘 + **튠마이뮤직(TuneMyMusic)** 소개
- 플랫폼 간 이동: 여러 스트리밍 서비스 간 플레이리스트/라이브러리 복사·전송
- 플레이리스트 생성: AI 프롬프트나 외부 텍스트/파일을 분석해 맞춤 재생목록 자동 생성

**발표 멘트**:
> "저희가 기획을 시작하며 먼저 살펴본 게 '튠마이뮤직(TuneMyMusic)'이라는 서비스입니다. 스포티파이, 유튜브뮤직, 애플뮤직처럼 서로 다른 스트리밍 플랫폼 사이에서 플레이리스트를 옮겨주고, 사용자가 입력한 텍스트나 프롬프트를 분석해서 재생목록을 자동으로 만들어주는 서비스입니다."

**대비 포인트**: 이 슬라이드는 아직 "기존 서비스 소개" 단계이고, 문제의식은 다음 페이지에서 이어집니다.

## 3p. 기획 배경 및 아이디어 ②

**화면 내용**:
- 문제의식 박스: **"아는 노래가 나왔을 때 즐거움"** — TuneMyMusic 같은 서비스는 사용자가 아는 노래 기반 추천도, 사용자화도 없음 → *"아는 노래가 없어서 못 듣겠어"*
- 화살표 → **사용자 친화적 앱 기획**: 사용자의 플레이리스트 곡을 추출한 뒤, 그 곡을 기반으로 플레이리스트 생성

**발표 멘트**:
> "그런데 튠마이뮤직 같은 서비스를 직접 써보니 한 가지 아쉬운 점이 있었습니다. 새로 만들어진 플레이리스트에 아는 곡이 없으면, 아무리 취향에 맞게 골랐다고 해도 결국 잘 안 듣게 되더라고요. '아는 노래가 없어서 못 듣겠다'는 게 저희가 찾은 핵심 문제였습니다. 그래서 저희는 사용자가 **이미 듣고 있는 플레이리스트에서 곡을 추출**하고, 그 곡을 기반으로 새 플레이리스트를 만들어주는 방향으로 기획을 잡았습니다."

**Q&A 대비**: "왜 새 곡 추천이 아니라 내 곡 기반인가?" → 위 스크립트 그대로 답변 가능.

## 4p. 웹 소개 (기능 요약표)

**화면 내용**: 주요 기능 8가지 표

| 기능 | 설명 |
|---|---|
| Spotify 로그인 | 사용자의 Spotify 계정으로 로그인 |
| 내 플레이리스트 불러오기 및 비율 조절 | 플레이리스트의 곡 불러오기 및 셔플 비율 설정 |
| 통합 검색 | 곡 및 가수 등 검색 가능 |
| 연쇄 탐색 | 곡을 선택하면 비슷한 곡으로, 아티스트를 선택하면 그 아티스트의 곡으로 화면 전환 |
| 무드·상황·장르 선택 | 사용자가 무드, 장르, 상황 등 선택 가능 |
| 재생 시간 맞춤 | 15분~2시간 중 원하는 시간을 고르면 시간에 맞게 생성 |
| 작명 기능 | 플레이리스트 이름 직접 입력 |
| Spotify 저장 | 완성된 목록을 실제 플레이리스트로 저장 |

**발표 멘트**:
> "저희 서비스의 핵심 기능을 한눈에 정리하면 이렇습니다. 로그인부터 저장까지 여덟 가지 기능이 있는데, 뒤에서 하나씩 화면과 함께 자세히 보여드리겠습니다."

**Q&A 대비**: 이 표는 목차 역할이므로, "지금부터 이 표 순서대로 하나씩 짚어드리겠다"고 예고하면 뒤 슬라이드와 자연스럽게 연결됩니다.

## 5p. 제작 과정

**화면 내용**: 5단계 카드 — **01 프로토타입 제작 → 02 Spotify API 연동 → 03 사용자 기반 웹 구현 → 04 UX 최적화 → 05 디자인 확정 및 Github 배포**

**발표 멘트**:
> "제작 과정은 크게 5단계로 진행했습니다. 먼저 프로토타입을 만들어 서비스 흐름을 검증하고, Spotify API를 연동한 뒤, 실제 사용자가 쓸 수 있는 형태로 웹을 구현했습니다. 이후 사용성을 다듬는 UX 최적화 과정을 거쳐, 최종 디자인을 확정하고 GitHub Pages로 배포했습니다."

**팁**: 이 슬라이드는 단계 이름만 나열되어 있어 짧게 넘어가도 되는 슬라이드입니다. 각 단계의 구체적인 내용(무엇을 했는지)은 6~10p 기능 슬라이드와 11p 문제 해결 슬라이드에서 자연스럽게 풀어주면 됩니다.

## 6p. 핵심 기능 ① — Spotify 연동 · 플레이리스트 불러오기 · 믹싱 비율

**화면 내용**: 좌측 텍스트 3개 + 우측 실제 화면 스크린샷 3장
- **사용자의 Spotify 연동**: Spotify 계정 그대로 로그인해 정보 불러오기
- **사용자의 playlist 불러오기**: 듣는 플레이리스트 불러오기 및 추출
- **믹싱 비율 설정**: 플레이리스트와 연관곡 믹싱 비율 설정
- 스크린샷: "Spotify로 로그인" 버튼 / "가져온 곡 70%에 관련 아티스트 추천곡 30%를 섞어 새 목록을 구성합니다" 안내문 + 플레이리스트 선택 드롭다운 / **믹스 비율 칩**(내 곡만·8:2·**7:3(기본)**·6:4·5:5)

**발표 멘트**:
> "먼저 로그인 화면입니다. 별도 회원가입 없이 'Spotify로 로그인' 버튼 하나로 시작합니다. 로그인하면 제 Spotify 플레이리스트를 불러올 수 있고, 여기서 중요한 게 **믹싱 비율 설정**입니다. 기본값은 제 곡 70%, 추천곡 30%인 7:3이지만, 내 곡만 듣고 싶으면 10:0으로, 새 곡을 더 많이 섞고 싶으면 5:5까지 직접 조절할 수 있습니다."

**Q&A 대비**: "추천곡은 어디서 가져오나요?" → "제 플레이리스트에 많이 등장하는 아티스트를 기준으로, 그 아티스트의 다른 곡들을 Spotify에서 가져와 섞습니다."

## 7p. 핵심 기능 ② — 검색과 연쇄 탐색

**화면 내용**: 좌측 텍스트 3개 + 우측 실제 화면 스크린샷(무드/상황/장르/가수 필터 칩 + 검색창 + 곡 목록 13곡)
- **사용자 플레이리스트 내 검색**: 이미 불러온 노래 목록 안에서 원하는 곡 수동 탐색
- **스포티파이 전체 곡 검색**: 플레이리스트 미선택 시 곡 검색으로 연관곡 플레이리스트 생성
- **플레이리스트 추출 후 연쇄적 탐색**: 목록에서 곡이나 가수를 누르면 비슷한 무드의 음악으로 부드럽게 화면 전환

**발표 멘트**:
> "검색은 두 가지 방식으로 동작합니다. 플레이리스트를 이미 불러왔다면 그 안에서 곡을 찾고, 아직 아무것도 안 불러왔다면 Spotify 전체 곡을 검색합니다. 그리고 저희가 신경 쓴 부분이 **연쇄 탐색**인데요, 목록에서 곡 하나를 누르면 비슷한 무드의 곡으로, 가수를 누르면 그 가수의 곡으로 화면이 자연스럽게 이어서 전환됩니다. 화면 오른쪽처럼 무드·상황·장르·가수 조건을 조합해서 원하는 곡만 걸러낼 수도 있습니다."

**Q&A 대비**: "비슷한 곡은 어떻게 판단하나요?" → 8p의 가중치 표로 자연스럽게 연결.

## 8p. 핵심 기능 ③ — 가중치 배정 방식 (추천 알고리즘)

**화면 내용**: 표

| 우선 분류 | 적용 규칙 | 스코어 가중치 |
|---|---|---|
| 선택 아티스트 | 동일 가수 곡 검출 | **+10점** |
| 선택한 무드 | 사용자 수동 일치 | **+4점** |
| 선택한 장르 | 사용자 수동 일치 | **+3점** |
| 추론 무드 | 상황 연계 자동 판정 | **+2점** |
| 추론 장르 | 무드 연계 자동 판정 | **+1점** |

**발표 멘트**:
> "곡을 추천할 때 저희는 단순 랜덤이 아니라 점수 기반으로 정렬합니다. 사용자가 직접 아티스트를 선택하면 가장 높은 10점, 무드를 직접 고르면 4점, 장르를 직접 고르면 3점을 줍니다. 그리고 사용자가 '상황'만 골라도, 저희 알고리즘이 그 상황에 어울리는 무드를 자동으로 추론해서 2점을, 그 무드에서 다시 장르를 추론해서 1점을 부여합니다. 이렇게 **사용자가 직접 고른 조건에 더 높은 가중치**를 주면서도, 조건을 적게 입력해도 알고리즘이 연관 조건을 자동으로 채워주는 구조입니다."

**Q&A 대비**: 이 슬라이드는 기술 심사에서 질문이 많이 나올 수 있는 부분입니다. "왜 이 가중치 값인가?" 같은 질문에는 "직접 선택한 조건일수록 사용자 의도가 명확하므로 더 높은 가중치를, 알고리즘이 추론한 조건은 보조적으로 낮은 가중치를 부여하는 원칙으로 설계했다"고 답하면 됩니다.

## 9p. 핵심 기능 ④ — 재생 시간 맞춤

**화면 내용**: 재생시간 칩(제한없음·15분·30분·45분·1시간·1시간30분·2시간) + 좌측 텍스트 3개
- **사용자가 시간 지정**: playlist 시간을 지정하면 시간에 맞게 생성
- **비율 유지**: 재생시간에 관계없이 (6p에서 설정한) 비율 유지
- **셔플 기능**: 플레이리스트를 생성할 때마다 곡을 무작위로 셔플

**발표 멘트**:
> "재생 시간도 직접 정할 수 있습니다. 15분부터 2시간까지 고르면, 앞서 설정한 믹싱 비율을 그대로 유지하면서 그 시간에 맞게 곡을 채워줍니다. 그리고 플레이리스트를 만들 때마다 곡 순서를 무작위로 섞어서, 같은 조건이라도 매번 조금씩 다른 결과를 받아볼 수 있습니다."

**Q&A 대비**: 11p에서 이 기능 관련 실제 버그 수정 사례(시간 설정 후 비율이 깨졌던 문제)가 나오므로, "처음엔 시간을 맞추다 보니 비율이 흐트러지는 문제가 있었는데, 이 부분은 뒤에서 자세히 말씀드리겠다"고 미리 언급하면 11p와 자연스럽게 이어집니다.

## 10p. 핵심 기능 ⑤ — 완성 및 Spotify 저장

**화면 내용**: 좌측 텍스트 2개 + 우측 완성 화면 스크린샷 (13곡 목록, 앨범 커버, "내 플레이리스트 · 2026. 7. 13." 이름, "이 플레이리스트를 Spotify에 저장" 버튼)
- **Spotify 연동**: 사용자의 Spotify로 생성된 playlist 전송
- **작명 기능**: 사용자가 플레이리스트 이름 입력 가능

**발표 멘트**:
> "모든 조건을 다 정하고 나면 이렇게 플레이리스트가 완성됩니다. 곡마다 앨범 커버와 무드·상황·장르 태그가 함께 표시되고, 상세 정보나 연관곡도 곡을 눌러 확인할 수 있습니다. 완성된 목록에는 이름을 직접 지어줄 수 있고, 이름을 안 정해도 조건 기반으로 자동으로 제안됩니다. 마지막으로 'Spotify에 저장' 버튼을 누르면 실제로 제 Spotify 계정에 이 플레이리스트가 그대로 생성됩니다."

**팁**: 가능하면 이 슬라이드에서 실제 라이브 데모(또는 짧은 영상)로 전환하면 임팩트가 가장 큽니다.

## 11p. 개발 과정에서 해결한 문제

**화면 내용**: 3개 사례 카드

| 단계 | 문제 | 해결 |
|---|---|---|
| 개발 테스트 단계 | 사용자 시간 설정 후 플레이리스트 제작 비율 오류 | 시간 설정 후 비율대로 곡을 새로 추천하도록 변경 |
| API 확보 단계 | 2026년 3월부터 Spotify가 웹 API 개발 모드 사용 시 계정에 **Spotify Premium 필수 구독**을 요구하도록 정책 변경 | Spotify Premium 구독을 통해 해결 |
| 개발 테스트 단계 | 가수 검색 시 같은 이름의 가수 노래가 무작위로 추천되는 문제 | 사용자가 가수 검색 후 가수를 직접 선택하는 기능 추가 |

**발표 멘트**:
> "개발하면서 실제로 겪었던 문제 세 가지를 공유드립니다. 첫째, 재생 시간을 맞추려다 보니 처음 설정한 믹싱 비율이 깨지는 문제가 있었는데, 시간에 맞춰 비율대로 곡을 다시 추천하도록 로직을 고쳐서 해결했습니다. 둘째, 개발 도중 Spotify 정책이 바뀌어서, 웹 API 개발 모드를 쓰려면 계정이 Spotify Premium을 구독하고 있어야 하는 상황이 생겼습니다. 실제로 저희가 Premium을 구독하고 나서야 정상적으로 개발을 이어갈 수 있었습니다. 셋째, 가수 이름으로 검색했을 때 동명이인 가수의 곡이 무작위로 섞여 나오는 문제가 있었는데, 검색 후 사용자가 원하는 가수를 직접 선택할 수 있는 기능을 추가해서 해결했습니다."

**⚠️ 발표 전 확인 사항**: 슬라이드에는 정책 변경 시점이 "2026년 3월"로 적혀 있습니다. 실제 Spotify 공식 발표일은 2026년 2월 11일로 확인되니, 발표 전에 어느 날짜가 맞는지 한 번 더 확인해보시는 걸 권장합니다(팀 내부적으로 실제 겪은 시점을 기준으로 하신 것일 수도 있어 그대로 두셔도 무방합니다).

**Q&A 대비**: 이 슬라이드가 심사위원/멘토에게 가장 좋은 인상을 줄 수 있는 파트입니다. "실제 서비스를 만들다 겪는 현실적인 문제(정책 변경, 엣지 케이스)에 어떻게 대응했는가"를 보여주는 슬라이드이므로, 각 문제에 대해 "왜 그 문제가 생겼는지" 한 문장씩 더 설명할 수 있게 준비해두면 좋습니다.

## 12p. 한계 및 발전 계획

**화면 내용**: 좌우 2단 카드

**한계 (왼쪽)**
- **스포티파이만 지원**: 스포티파이 미사용 시 사용 불가
- **비공개 테스트 제한 규정**: 개발자 모드 규약으로 인해 승인 인원 최대 5인 제약
- **가사 데이터 미지원**: 무료 API 라이선스 문제로 직접 곡 가사 검색 불가

**발전 계획 (오른쪽)**
- **다른 음악 앱 API 연동**: 유튜브뮤직, 멜론 등의 API 연동
- **공식 심사 신청 및 해제**: Extended Quota 심사를 거쳐 사용 가능 사용자 수 증가
- **외부 가사 연동**: Musixmatch 등 API 연동

**발표 멘트**:
> "현재 저희 서비스의 한계도 솔직하게 말씀드리겠습니다. 지금은 스포티파이 계정이 있어야만 사용할 수 있고, 개발자 모드 규약상 최대 5명까지만 승인해서 쓸 수 있습니다. 또 가사 데이터는 무료 API 라이선스 문제로 직접 검색할 수 없습니다. 이런 한계를 해결하기 위해 앞으로 유튜브뮤직이나 멜론 같은 다른 음악 앱과도 연동하고, Spotify의 Extended Quota 심사를 신청해서 사용 가능한 인원을 늘리고, Musixmatch 같은 외부 가사 API를 연동할 계획입니다."

**Q&A 대비**: 한계를 먼저 인정하고 바로 발전 계획으로 답을 준비해뒀다는 흐름 자체가 설득력 있는 답변이 됩니다. "그럼 언제쯤 다른 사람도 쓸 수 있나요?" → "Spotify의 Extended Quota Mode 심사를 통과하면 가능한데, 그 전까지는 팀원/멘토 등 최대 5명 계정으로 시연 가능합니다."
