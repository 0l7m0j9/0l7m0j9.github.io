# Playlist Maker — 발표 진행 가이드

`Playlist Maker Presentation.pdf` (전체 11장, 표지 포함) 기준 스피치 가이드입니다.
총 발표 시간을 8~10분으로 가정해 슬라이드별 소요 시간을 배분했습니다. 팀 인원수에 맞춰 구간을 나눠 맡으면 됩니다.

---

## 발표 전 체크리스트 (먼저 확인)

PDF를 코드/원본 자료와 대조했을 때 발견된 부분입니다. 발표 전에 고치거나, 못 고치면 발표 중 말로 자연스럽게 덮을 수 있도록 표시해뒀습니다.

- [ ] **4번 슬라이드 날짜 오류**: "2026년 3월부터" → 정확히는 **2026년 2월 11일**. 슬라이드 수정이 안 되면 말할 때 "2월 중순"으로 자연스럽게 언급
- [ ] **4번 슬라이드 빈 괄호**: "바뀐 전송 경로(에서 )" — 괄호가 비어있음. 발표 중 이 문장은 화면에 띄우지 말고 말로만 "`/tracks`가 `/items`로, `/users/{id}/playlists`가 `/me/playlists`로 바뀌었다"고 설명
- [ ] **기술 구조 슬라이드가 아예 없음**: 정적 파일 1개 / 서버 비용 0원 / PKCE를 쓸 수밖에 없었던 이유가 슬라이드에 없으므로, **6번 슬라이드(로그인) 발표 시 구두로 반드시 보충** (아래 6번 항목 참고)
- [ ] 8번 슬라이드 가중치 표에 "가수 선택 시 그 가수의 다른 장르 곡 +3점" 행이 빠져있음 — 질문 나오면 답할 수 있게 숙지만 해두기 (표 자체를 급히 고칠 필요는 없음)

## 슬라이드별 가이드

### 1. 표지 (0:00~0:20, 20초)
**보여주며**: 로고/이퀄라이저 장식만 짧게
**말할 내용**: "저희 4조는 Spotify 기반으로 개인 맞춤 플레이리스트를 만들어주는 웹 서비스, Playlist Maker를 만들었습니다."
**팁**: 표지에서 길게 끌지 말 것. 청중 시선을 화면에서 발표자로 옮기는 짧은 워밍업 구간.

### 2. 기존 서비스 비교 (0:20~1:10, 50초)
**말할 내용**: 좌측 카드(기존 서비스)를 먼저 짚고, "저희는 반대로 접근했습니다"로 우측 카드(Playlist Maker)로 넘어가기.
**강조 문장**: "기존 도구들은 '내가 모르는 곡'을 추천하는 데 초점이 맞춰져 있는데, 저희는 '내가 이미 듣는 곡'에서 출발합니다."
**팁**: 표를 읽지 말고 좌→우 대비만 짚기. 원곡:추천곡 비율(10:0~5:5)은 다음 기능 슬라이드에서 다시 나오니 여기선 가볍게.

### 3. 제작 과정 로드맵 (1:10~2:00, 50초)
**말할 내용**: 5단계 카드를 순서대로 짚되, 1번(Fake Door)과 2번(Spotify 전환)만 조금 더 설명.
**강조 문장**: "처음엔 실제 API 연결 없이 화면 흐름만 가짜로 만들어서 검증했고, 한계를 느낀 뒤 실제 Spotify 연동으로 전면 개편했습니다."
**팁**: 나머지 3~5번(구축/UX/배포)은 "이후 기능을 다듬고 배포까지 마쳤습니다" 한 문장으로 압축 가능. 시간 없으면 이 슬라이드에서 가장 먼저 줄일 것.

### 4. 트러블슈팅 (2:00~3:00, 60초) — 스토리텔링 핵심 슬라이드
**말할 내용**: 3개 박스를 순서대로.
1. "시간 설정을 먼저 하도록 UI 순서를 바꿔서 비율 오류를 해결했고"
2. "2026년 2월, Spotify가 개발자 정책을 크게 바꿨습니다. 앱 소유자가 Premium을 구독해야 하고, 로그인 인원도 5명으로 제한됐어요. 이때 API 엔드포인트 경로도 바뀌어서(`/tracks`→`/items` 등) 기존 코드가 갑자기 403을 반환했고, 원인을 추적해서 새 경로로 마이그레이션했습니다."
3. "같은 이름의 아티스트가 섞여 나오는 문제는 사용자가 직접 아티스트를 선택하는 기능을 추가해 해결했습니다."
**팁**: 이 슬라이드가 "진짜 개발 경험"을 보여주는 슬라이드입니다. 발표에서 가장 힘주어 말할 구간 — 실제 서비스 정책이 바뀌는 걸 대응한 경험은 교수/청중이 좋아하는 포인트.

### 5. 사용자 여정 (3:00~3:50, 50초)
**말할 내용**: 우측 스크린샷을 손으로 짚어가며 4단계를 순서대로. "로그인 → 내 목록 불러오고 비율 설정 → 재생시간·조건 선택 → 저장" 흐름을 실제 화면 순서 그대로 설명.
**팁**: 가능하면 이 슬라이드에서 라이브 데모로 전환하거나, 데모 영상을 짧게 트는 것도 좋음. 시간 여유 있으면 여기서 실제 서비스를 잠깐 보여주는 게 임팩트가 큼.

### 6. Core Feature 01 — 로그인 & 비율 조절 (3:50~4:50, 60초)
**말할 내용**: 로그인 카드 → 비율 슬라이더 순서로.
**⚠️ 여기서 빠진 기술 구조 내용 구두 보충**: "이 로그인은 서버 없이 브라우저에서만 동작합니다. 원래 OAuth는 Client Secret이 필요한데, 저희는 서버가 없어서 Secret을 안전하게 보관할 곳이 없습니다. 그래서 Secret이 필요 없는 PKCE 방식을 썼고, 덕분에 정적 파일 하나로 GitHub Pages에 무료로 배포할 수 있었습니다."
**팁**: 이 한 문단이 원래 있어야 했던 "기술 구조" 슬라이드 내용을 대신합니다. 반드시 언급할 것 — 서버 없이 동작한다는 게 이 프로젝트의 기술적 핵심 포인트입니다.

### 7. Core Feature 02 — 통합 검색 (4:50~5:40, 50초)
**말할 내용**: "제 플레이리스트 안에서든, Spotify 전체에서든 검색이 가능하고, 곡이나 아티스트를 누르면 비슷한 무드의 곡으로 화면이 전환됩니다."
**팁**: "완벽한 AI 추천은 아니고, 아티스트·장르 조합으로 근사한 것"이라는 한계는 11번 슬라이드에서 다루니 여기선 굳이 먼저 말하지 않아도 됨.

### 8. Core Feature 03 — 무드/상황 매칭 (5:40~6:40, 60초) — 표 있는 슬라이드
**말할 내용**: 우측 예시(카페 → 여유로운 → R&B)를 먼저 보여준 뒤, 좌측 표로 "이 추론과 직접 선택을 점수로 합산해서 정렬한다"고 설명.
**핵심 한 문장**: "사용자가 직접 고른 조건일수록 점수를 높게, 알고리즘이 자동으로 추측한 조건일수록 낮게 줘서 — 항상 사용자 의도가 우선하도록 설계했습니다."
**예상 질문 대비**: "가수를 골랐는데 그 가수 곡이 없으면요?" → "그 가수와 같은 장르의 다른 곡에도 낮은 점수(+3점)를 줘서 결과가 비게 되는 걸 막았습니다." (표에는 없지만 실제 로직에 있는 부분 — 질문 오면 이렇게 답변)

### 9. Core Feature 04 — 재생시간 & 셔플 (6:40~7:20, 40초)
**말할 내용**: "15분부터 2시간까지 고르면 그 시간에 맞게 곡 수가 자동으로 채워지고, 비율은 그대로 유지됩니다. 만들 때마다 순서는 무작위로 섞여요."
**팁**: 짧게 가는 슬라이드. 버튼 UI를 손으로 가리키며 빠르게 진행.

### 10. Core Feature 05 — 결과 화면 (7:20~8:00, 40초)
**말할 내용**: "완성된 목록은 실제 Spotify 앨범 커버로 표시되고, 이름을 안 정하면 선택한 무드로 자동 제안됩니다. 저장 버튼 누르면 실제 제 Spotify 앱에 바로 생깁니다."
**팁**: "실제 제 계정에 생긴다"는 점을 강조 — 이 프로젝트가 목업이 아니라 실제로 동작한다는 걸 보여주는 슬라이드.

### 11. 한계 & 개선 계획 (8:00~9:00, 60초)
**말할 내용**: 좌측(한계)을 먼저 담백하게 인정 → 우측(계획)으로 자연스럽게 연결.
**강조 문장**: "오디오 특성 API와 추천 API가 개발 중에 막혀서, 저희 결과물은 완벽한 AI 분석이 아니라 근사치입니다. 이 부분은 솔직하게 말씀드립니다."
**팁**: 한계를 숨기지 말고 담담하게 말하는 게 오히려 신뢰도를 높임. "그래서 다음 단계로 이런 걸 계획하고 있습니다"로 마무리.

## 예상 Q&A 대비

**Q. 서버 없이 어떻게 로그인/인증이 되나요?**
A. OAuth 2.0의 PKCE 방식을 씁니다. 원래 필요한 Client Secret 없이, 브라우저가 임시 코드(code_verifier/code_challenge)를 만들어 Spotify와 직접 주고받습니다. Secret을 안전하게 보관할 서버가 없어서 이 방식을 선택했습니다.

**Q. 진짜 AI 추천인가요?**
A. 아닙니다. Spotify의 공식 추천(Recommendations) API가 정책상 막혀서, 아티스트·장르 조합 기반 규칙(점수 가중치)으로 근사한 것입니다. 이 부분은 한계 슬라이드에서 명확히 밝혔습니다.

**Q. 왜 최대 5명만 쓸 수 있나요?**
A. Spotify Development Mode 정책 때문입니다. 앱 심사를 받지 않은 상태에서는 로그인 허용 인원이 5명으로 제한됩니다. 확장하려면 Spotify의 Extended Quota Mode 심사를 받아야 합니다.

**Q. 개발 중 제일 힘들었던 부분은?**
A. 2026년 2월, Spotify가 개발자 정책과 엔드포인트를 동시에 바꿔서 멀쩡히 되던 기능이 갑자기 403 에러를 낸 것이었습니다. 원인을 하나씩 추적해서 새 엔드포인트로 마이그레이션했습니다.

**Q. 가중치 점수는 어떻게 정했나요?**
A. 사용자가 직접 선택한 조건(가수·무드·장르)에는 높은 점수를, 알고리즘이 상황이나 무드에서 자동으로 유추한 조건에는 낮은 점수를 줬습니다. 그래야 사용자가 원한 것이 항상 우선순위에 오도록 만들 수 있었습니다.
