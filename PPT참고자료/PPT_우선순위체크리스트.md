# Playlist Maker — 발표 내용 우선순위 체크리스트

`PPT_참고자료.md`(13페이지 발표 가이드)의 내용을 색으로 구분했습니다. 시간이 부족하면 🔴만 말해도 발표가 성립하고, 시간이 남으면 🟡·🟢 순서로 살을 붙이면 됩니다.

## 범례

- 🔴 **필수** — 이게 빠지면 발표 흐름이 끊기거나 핵심 차별점이 전달 안 됨
- 🟡 **있으면 좋음** — 설득력·완성도를 높여주지만 시간 없으면 생략 가능
- 🟢 **선택** — 청중 성격(기술 심사 vs 일반 청중)이나 질문에 따라 골라서 사용

---

## 1p. 표지

- 🔴 서비스명 + 한 줄 정의("내 음악 취향, 무드, 조건에 맞춰 새 노래를 믹싱하는 개인 맞춤 웹 서비스")
- 🟡 "김포시-연세대 AI·SW 동아리 멘토링" 소속 프로그램 언급
- 🟢 이퀄라이저 그래픽 등 디자인 요소에 대한 설명

## 2p. 기획 배경 및 아이디어 ①

- 🔴 벤치마킹 사례로 TuneMyMusic 언급 (경쟁 서비스를 분석했다는 근거)
- 🟡 TuneMyMusic의 두 가지 핵심 기능(플랫폼 간 이동 / AI 기반 생성) 구분 설명
- 🟢 화면의 Spotify·유튜브뮤직·Apple Music·FLO 아이콘 하나하나 짚어가며 설명

## 3p. 기획 배경 및 아이디어 ②

- 🔴 "아는 노래가 없어서 못 듣겠어" — 핵심 문제의식 (가장 중요한 차별화 근거이므로 반드시 힘주어 전달)
- 🔴 해결 방향: 사용자 플레이리스트 곡 추출 → 그 곡 기반 플레이리스트 생성
- 🟡 인용구("아는 노래가 없어서...")를 실제 사용자 반응처럼 실감나게 전달

## 4p. 웹 소개 (기능 요약표)

- 🔴 8개 기능을 표 순서대로 한 줄씩 빠르게 언급 (전체 그림을 먼저 보여주는 목차 역할)
- 🟡 "지금부터 이 표 순서대로 하나씩 화면과 함께 보여드리겠다"는 예고 멘트
- 🟢 이 단계에서 기능을 상세히 설명하지 않기 — 뒤 슬라이드에서 다룰 내용이므로 여기선 길게 끌지 않는 것을 권장

## 5p. 제작 과정

- 🟡 5단계(프로토타입 → API 연동 → 웹 구현 → UX 최적화 → 배포) 이름을 순서대로 언급
- 🟢 각 단계에서 구체적으로 무엇을 했는지 상세 설명 (시간이 부족하면 이 슬라이드는 5초 내로 넘겨도 무방)

## 6p. 핵심 기능① — 로그인 · 불러오기 · 믹싱 비율

- 🔴 Spotify 로그인으로 시작한다는 것
- 🔴 믹싱 비율을 사용자가 직접 정할 수 있다는 것(10:0~5:5) — 가장 눈에 띄는 차별 기능이므로 화면 스크린샷과 함께 꼭 강조
- 🟡 추천곡을 어디서 가져오는지(내 플레이리스트에 많이 등장하는 아티스트 기반) 근거 설명
- 🟢 PKCE, OAuth 같은 인증 방식 기술 용어까지 설명 (기술 심사 자리면 🟡로 격상)

## 7p. 핵심 기능② — 검색과 연쇄 탐색

- 🔴 연쇄 탐색(곡 선택→비슷한 곡, 가수 선택→그 가수 곡으로 화면 전환) — 독창적 UX이므로 반드시 강조
- 🟡 두 가지 검색 모드(불러온 목록 내 검색 vs Spotify 전체 검색) 구분 설명
- 🟢 무드·상황·장르·가수 필터 칩 UI 디자인 디테일 설명

## 8p. 핵심 기능③ — 가중치 배정 방식

- 🔴 "단순 랜덤이 아니라 점수 기반 알고리즘으로 정렬한다"는 사실 자체
- 🟡 구체적 점수(아티스트+10, 무드+4, 장르+3, 추론무드+2, 추론장르+1) 하나하나 언급
- 🟢 왜 이런 가중치 값으로 설계했는지 설계 근거 — 질문 들어왔을 때 답변용으로 아껴두기

## 9p. 핵심 기능④ — 재생 시간 맞춤

- 🔴 원하는 재생 시간을 고르면 그 시간에 맞게 자동 생성된다는 것
- 🔴 재생시간을 바꿔도 앞서 정한 믹싱 비율은 유지된다는 것 (6p와 이어지는 핵심 연결 고리)
- 🟡 매번 곡 순서를 무작위로 셔플한다는 점
- 🟢 15분~2시간 선택지를 전부 하나하나 나열

## 10p. 핵심 기능⑤ — 완성 및 Spotify 저장

- 🔴 "Spotify에 저장" 버튼으로 실제 내 계정에 진짜 플레이리스트가 생긴다는 것 (가장 강력한 "실제로 동작한다"는 증거이므로 반드시 강조, 가능하면 라이브 데모)
- 🟡 이름을 직접 짓거나 자동 제안받을 수 있다는 작명 기능
- 🟢 앨범 커버, 무드/상황/장르 태그 표시 등 결과 화면의 디테일

## 11p. 개발 과정에서 해결한 문제

- 🔴 Spotify Premium 필수 구독 정책 변경 대응 사례 — 실제 서비스 운영의 현실을 보여주는 가장 설득력 있는 스토리이므로 반드시 포함
- 🟡 나머지 두 사례(재생시간-비율 오류, 가수 동명이인 문제)도 언급
- 🟢 각 문제가 왜 발생했는지 기술적 원인까지 상세 설명 (질문 대비용으로 준비만 해두기)

> ⚠️ 발표 전 확인: 이 페이지의 "2026년 3월" 날짜가 실제 겪은 시점이 맞는지 한 번 더 체크해보세요(Spotify 공식 발표일은 2026년 2월 11일로 조사됨).

## 12p. 한계 및 발전 계획

- 🔴 한계를 최소 1개 이상 정직하게 인정 (신뢰도를 높이는 파트이므로 생략하지 않기)
- 🔴 그에 대응하는 발전 계획을 최소 1개 이상 제시 (한계로만 끝내지 않고 비전으로 마무리)
- 🟡 한계 3개, 발전 계획 3개를 전부 짝지어 설명
- 🟢 발전 계획의 구체적 API명(Musixmatch 등)까지 언급

## 13p. 감사합니다

- 🔴 마무리 인사 + 질문 받겠다는 멘트
- 🟢 질문이 없을 경우를 대비한 추가 마무리 코멘트(예: 데모 링크 재안내)

## 발표 시간별 추천 시나리오

| 상황 | 전략 |
|---|---|
| **시간이 매우 부족함 (3~4분)** | 🔴만 전부 말하기. 5p는 통째로 스킵 가능 |
| **표준 발표 (7~8분)** | 🔴 + 🟡 전부, 🟢는 질문 나오면 답변 |
| **시간 넉넉함 / 기술 심사 (10분 이상)** | 🔴🟡🟢 전부 포함, 8p·11p는 🟢까지 상세히 |
