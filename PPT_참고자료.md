## 슬라이드 1. 기존 서비스와 Playlist Maker 비교

**핵심 메시지**: 기존 플레이리스트 추천 서비스는 "새 곡 추천"에만 초점 — Playlist Maker는 "내가 이미 듣는 곡"을 출발점으로 삼는다.

| 구분 | 기존 서비스 | Playlist Maker |
|---|---|---|
| 출발점 | 알고리즘이 고른 낯선 곡 | 내 Spotify 플레이리스트 |
| 새 곡 비중 | 대부분/전부 새 곡 | 사용자가 직접 정하는 비율 (내 곡만 ~ 5:5) |
| 조건 설정 | 장르 정도로 단순 | 무드·상황·장르·가수·재생 시간까지 조합 |
| 곡 순서 | 고정 | 만들 때마다 무작위 셔플 |
| 탐색 방식 | 목록 스크롤 위주 | 곡 선택 시 "비슷한 곡", 아티스트 선택 시 "그 아티스트 곡"으로 즉시 전환 |

**스피커 노트**: "이미 듣는 곡에서 딱 맞는 새 플레이리스트를"이라는 한 문장으로 기획 의도를 요약. 완전히 새로운 곡 추천 서비스가 아니라, **내 취향을 우선하고 필요한 만큼만 새 곡을 섞는** 서비스라는 점을 강조.

## 슬라이드 2. Playlist Maker 제작 과정

**핵심 메시지**: 가짜 화면으로 방향을 먼저 검증한 뒤, 실제로 되는 것만 남기고 전면 재구축했다.

**단계별 흐름**:

1. **기획 — Fake Door MVP** (실제 API 연동 없이 화면 흐름만 검증)
   - YouTube 플레이리스트 링크 입력 UI
   - Spotify/Apple Music/지니/멜론/FLO 등 여러 음악 앱 "연동인 척"하는 버튼
   - 마이크로 음악 인식 (가짜 시뮬레이션)
   - 가사로 곡 찾기 (미리 준비한 가짜 가사 DB)
   - 샘플 곡 30~50곡에 무드/상황/장르 태그를 수동으로 미리 붙여둠
   - 목적: 실제 API 연동 전에 "사용자가 이런 흐름을 자연스럽게 느끼는가"를 먼저 검증
2. **전환 결정 — 왜 실제 연동으로 바꿨나**
   - Fake Door로는 검증에 한계 (실제 내 플레이리스트가 아니라 샘플 데이터라 몰입감이 떨어짐)
   - "그냥 실제로 되게 만들자"는 방향으로 전면 개편 결정
3. **Spotify 실제 연동 구축**
   - OAuth 2.0 PKCE 로그인 구현 (서버 없이)
   - 내 플레이리스트 불러오기 + 추천곡 믹스 로직 구현
   - YouTube 링크 입력, 여러 플랫폼 fake 연동, 마이크 인식, 가사 검색 **전부 제거** (실제로 안 되는 기능을 흉내내는 건 의미 없다고 판단)
4. **UX 반복 개선** (사용자 피드백 기반)
   - 검색 기능을 "무드별 추천 안에 숨어있던 위치"에서 **메인 화면**으로 이동
   - 재생 시간 선택 위치를 화면 하단 → 상단(설정값이므로 먼저 정하게)
   - 원곡:추천곡 고정 비율(7:3) → **사용자가 직접 선택**하는 비율로 변경
   - 검색 결과가 "같은 제목 곡만 나열"되는 문제 발견 → 텍스트 검색과 장르 필터 분리, 중복 제거 로직 추가
   - 곡 셔플이 항상 비슷한 순서로 나온다는 피드백 → 결과 순서 매번 무작위화
5. **브랜딩 & 문서화**
   - 로고/파비콘 적용
   - README, 발표 자료 정리

**스피커 노트**: 처음부터 완벽한 설계가 아니라 **가짜 화면 → 실제 연동 → 반복 피드백 개선**이라는 순서로 만들어졌다는 점이 이 슬라이드의 핵심.

## 슬라이드 3. Playlist Maker 제작 과정 및 난관

**핵심 메시지**: 개발 도중 Spotify 자체 정책이 계속 바뀌면서, 이미 만든 기능이 갑자기 멈추는 상황을 여러 번 겪고 그때그때 대응했다.

**타임라인**:

| 시점 | 무슨 일이 있었나 | 어떻게 대응했나 |
|---|---|---|
| 2024~2026 (누적) | 오디오 특성(BPM·에너지) API, 추천(Recommendations) API, 아티스트 배치 조회 등이 단계적으로 폐지됨 | 오디오 분석 기반 정밀 태깅 대신, 아티스트 실제 장르 정보 + 텍스트 키워드 휴리스틱으로 무드/장르를 자체 추정하는 로직으로 설계 전환 |
| 2026.02.11 | Spotify Development Mode 정책 대개편 발표 — 앱 소유자 **Premium 계정 필수**, 앱 1개 제한, 로그인 최대 5명 제한 | 실제로 Premium을 구독하고 나서야 앱 생성이 가능했음. 개발을 하다가 정책이 바뀌어서, 이미 짜둔 로그인 UI를 다시 점검해야 했음 |
| 2026.02.11 (같은 정책) | 엔드포인트 경로 변경 — `/playlists/{id}/tracks`→`/items`, `/users/{id}/playlists`→`/me/playlists`, `GET /artists`(배치) 폐지 | 기존 코드가 갑자기 403 / 빈 응답을 반환 → 하나씩 원인 추적 후 새 경로로 마이그레이션, 배치 조회는 아티스트별 개별 호출로 대체 |
| 개발 중 발견 | 무드/상황 칩과 검색어를 같이 Spotify에 보내면 **엉뚱한 곡**(제목은 안 맞고 장르만 맞는 곡)이 섞여 나옴 | 검색어가 있으면 텍스트만으로 검색하고, 무드/상황 필터는 결과 안에서 다시 걸러내도록 분리 |
| 개발 중 발견 | 같은 곡의 싱글판·리마스터판이 다른 트랙 ID로 중복 나열됨 | 트랙 ID뿐 아니라 "제목+아티스트" 기준으로 한 번 더 중복 제거 |
| 개발 중 발견 | 재생 시간을 채우려 해도 추천곡이 부족함 | 추천곡을 처음부터 필요량의 몇 배(최대 80곡)로 넉넉히 가져와 저장해두고, 재생시간 채울 때 그 풀에서 추가로 끌어오도록 개선 |

**스피커 노트**: 이 슬라이드가 발표에서 가장 설득력 있는 파트. "완성된 걸 보여주는" 발표가 아니라 **실제 서비스 정책 변화에 대응하며 만든 살아있는 프로젝트**라는 인상을 줌. 시간이 부족하면 2026.02.11 정책 변화 한 줄만 강조해도 충분히 임팩트 있음.

## 슬라이드 4. Playlist Maker 기술 구조

**핵심 메시지**: 서버 없이, 정적 파일 하나로 Spotify와 브라우저가 직접 통신한다.

**구조 다이어그램(텍스트로 표현)**:
```
[브라우저 (index.html)] ──── PKCE OAuth ────> [Spotify 로그인 화면]
[브라우저 (index.html)] <──── 토큰 ─────────  accounts.spotify.com
[브라우저 (index.html)] ──── API 호출 ──────> api.spotify.com (Web API)
```
백엔드 서버가 없어서 중간에 우리 서버를 거치지 않고, **사용자의 브라우저가 Spotify와 직접 통신**합니다.

**구성 요소**:

| 구분 | 내용 |
|---|---|
| 구조 | 단일 정적 파일(`index.html`) — HTML/CSS/JS 전부 포함, 빌드 도구 없음 |
| 인증 | OAuth 2.0 Authorization Code + PKCE (Client Secret 불필요) |
| 토큰 저장 | `localStorage`(Access/Refresh Token), `sessionStorage`(PKCE verifier) |
| 데이터 소스 | Spotify Web API만 사용, 샘플/더미 데이터 없음 |
| 호스팅 | GitHub Pages — `사용자명.github.io` 저장소, `main` 브랜치 푸시로 자동 배포 |

**실제 호출 엔드포인트**:
- `GET /me/playlists` — 내 플레이리스트 목록
- `GET /playlists/{id}/items` — 플레이리스트 트랙 목록
- `GET /search` (`type=track` 또는 `type=track,artist`) — 곡/아티스트 검색
- `GET /artists/{id}` — 아티스트 장르 정보 (개별 호출)
- `POST /me/playlists` — 새 플레이리스트 생성
- `POST /playlists/{id}/items` — 플레이리스트에 곡 추가

**스피커 노트**: "서버 비용 0원", "배포가 git push 한 줄"이라는 점을 강조하면 심플함이 잘 전달됨. 동시에 서버가 없기 때문에 Client Secret을 못 숨긴다는 제약 → 그래서 PKCE를 쓸 수밖에 없었다는 인과관계로 연결하면 자연스러움.

## 슬라이드 5. 사용자의 Playlist Maker 사용 방법

**핵심 메시지**: 로그인부터 저장까지 5단계, 별도 회원가입이나 앱 설치 없이 브라우저에서 끝난다.

**사용 흐름 (사용자 시점 단계별)**:

1. **로그인** — "Spotify로 로그인" 버튼 클릭 → Spotify 자체 로그인 화면에서 이메일/비밀번호 입력 → 우리 사이트로 자동 복귀
2. **곡 채우기 — 두 가지 방법 중 선택**
   - (A) 내 플레이리스트 선택 → 믹스 비율(예: 7:3) 선택 → 불러오기
   - (B) 아직 아무것도 없는 상태에서 검색창에 곡/아티스트 입력 → Spotify 카탈로그 검색 결과가 바로 표시
3. **조건으로 좁히기** — 무드·상황·장르·가수 칩을 클릭하면 조건에 맞는 곡만 추려지고, 재생 시간(15분~2시간)을 고르면 그 시간에 맞게 곡 수가 자동 조정됨
4. **곡 하나 눌러 더 탐색하기** — 검색 결과에서 곡을 선택하면 "비슷한 곡"으로, 아티스트를 선택하면 "그 아티스트의 곡"으로 화면이 전환됨 (계속 파고들며 탐색 가능)
5. **완성 & 저장** — "플레이리스트 만들기" → 결과 확인 → 이름을 직접 입력(또는 자동 제안 이름 사용) → "Spotify에 저장" → 내 Spotify 앱에서 바로 확인 가능

**스피커 노트**: 이 슬라이드는 실제 데모 스크린샷이나 화면 흐름 GIF를 곁들이면 가장 효과적입니다. 발표 시간이 있다면 이 슬라이드에서 라이브 데모로 전환하는 것도 좋습니다.

## 슬라이드 6. Playlist Maker 기능 ① — Spotify 로그인 & 플레이리스트 불러오기

- **PKCE 로그인**: Client Secret 없이 브라우저에서 안전하게 인증. 로그인 버튼 → Spotify 공식 로그인 화면 → 자동 복귀
- **로그인 유지**: 토큰을 `localStorage`에 저장해 새로고침해도 로그인 상태 유지 (단, 불러온 곡 목록은 새로고침 시 초기화 — 다시 "불러오기" 필요)
- **믹스 비율 커스터마이즈**: 내 곡만(10:0) / 8:2 / **7:3(기본)** / 6:4 / 5:5 중 선택
  - 비율을 바꿔도 API 재호출 없이, 이미 가져온 곡 풀로 즉시 재구성
  - 추천곡은 내 플레이리스트 상위 아티스트 12명 × 최대 10곡, 최대 80곡까지 미리 넉넉히 확보
- **자연스러운 믹스**: 원곡 순서는 유지하면서, 추천곡을 정해진 비율로 가중치 랜덤 방식으로 끼워 넣음 (기계적으로 균등 배치하지 않음)

## 슬라이드 7. Playlist Maker 기능 ② — 통합 검색 (비슷한 곡 · 아티스트 선택)

- **두 가지 검색 모드**: 플레이리스트를 불러온 상태면 그 안에서 로컬 검색, 안 불러온 상태면 **Spotify 카탈로그를 직접 검색**
- **곡 선택 → 비슷한 곡**: 검색 결과에서 곡 하나를 클릭하면, 같은 아티스트의 다른 곡 + 같은 장르(Spotify 실제 장르 정보 기준)의 곡을 모아 "비슷한 곡" 목록으로 전환
- **아티스트 검색 → 아티스트 선택**: 검색어가 아티스트명과 겹치면 아티스트 후보 칩이 함께 뜨고, 클릭하면 그 아티스트의 곡들로 전환
- **검색 정확도 개선**: 검색어가 있을 때는 무드/상황 필터를 검색 쿼리에 섞지 않고 텍스트로만 검색 (섞으면 엉뚱한 곡이 나오는 문제가 있었음)
- **중복 제거**: 트랙 ID + 제목/아티스트 기준 이중 중복 제거로 같은 곡의 여러 버전이 반복 노출되는 문제 해결
- **한계**: Spotify의 공식 추천(Recommendations) API가 막혀 있어, 진짜 AI 추천이 아니라 아티스트·장르 조합으로 근사한 결과입니다

## 슬라이드 8. Playlist Maker 기능 ③ — 무드 · 상황 · 장르 추론 알고리즘

- **계층 추론**: 상황을 고르면 연관 무드를, 무드를 고르면 연관 장르를 자동 추론
  - 예: `상황 "카페"` → `무드: 여유로운·따뜻한·설레는` → `장르: 힙합·R&B·인디`
- **점수 기반 정렬**: 가수 직접선택 +10 / 무드 직접선택 +4 / 무드 추론 +2 / 장르 직접선택 +3 / 장르 추론 +1점을 합산해 정렬
- **UI로 추론 과정을 투명하게 공개**: 직접 선택한 칩은 실선, 알고리즘이 자동으로 딸려온 연관 항목은 점선("inferred")으로 구분 표시. 화면에 "상황→무드→장르" 추론 경로를 화살표로 함께 보여줌
- **태깅 방식**: 오디오 특성 API가 막혀 있어, 아티스트 실제 장르 정보 + 텍스트 키워드 + 곡 ID 기반 다양성 보정을 조합해 무드/장르를 추정 (완벽한 정답이 아니라 근사치)

## 슬라이드 9. Playlist Maker 기능 ④ — 재생 시간 맞춤 & 결과 셔플

- **재생 시간 선택**: 15분 / 30분 / 45분 / 1시간 / 1시간 30분 / 2시간 / 제한 없음
- **비율을 지키며 시간 채우기**: 선택한 믹스 비율(예: 5:5)을 유지한 채 목표 시간까지 원곡·추천곡 풀에서 채움. 화면에 즉시 반영(별도 버튼 없이 실시간 재계산)
- **결과 셔플**: "플레이리스트 만들기"를 누를 때마다 최종 곡 순서를 무작위로 재배치 — 필터링/추천 로직은 그대로 두고 **순서만 매번 새롭게**
- **저장 일관성**: Spotify에 저장할 때도 화면에 보여준 셔플된 순서 그대로 반영

## 슬라이드 10. Playlist Maker 기능 ⑤ — 결과 화면 (앨범 커버 · 이름 입력 · 저장)

- **앨범 커버 표시**: 곡 목록, 결과 카드, 상세 모달, 연관곡까지 전부 실제 Spotify 앨범 커버 이미지 표시
- **플레이리스트 이름 직접 입력**: 결과 화면에서 이름을 직접 타이핑하거나, 비워두면 선택 조건 기반으로 자동 제안(예: "카페 플레이리스트 · 2026.07.13")
- **Spotify에 실제 저장**: "Spotify에 저장" 버튼 → 내 Spotify 계정에 실제 플레이리스트 생성, 새 탭으로 자동으로 열림, 저장 진행 상황(몇 곡 중 몇 곡) 표시
- **더 보기 페이지네이션**: 곡이 많을 때 20곡씩 "더 보기"로 펼쳐 스크롤 부담을 줄임

## 슬라이드 11. Playlist Maker 한계

1. **태깅은 근사치** — Spotify가 오디오 특성(BPM·에너지) API를 폐지해, 무드/장르는 완벽한 AI 분석이 아닌 추정치
2. **"비슷한 곡"도 근사치** — 공식 추천 API가 막혀 아티스트·장르 조합으로 대체 (진짜 추천 엔진 아님)
3. **가사 검색·마이크 인식 미지원** — Spotify Web API가 가사 데이터를 아예 제공하지 않음
4. **최대 5명 제한** — Development Mode 제약: 앱 1개, 로그인 최대 5명, 개발자 Premium 필수
5. **공개 서비스로는 아직 부적합** — 불특정 다수 대상 서비스로 넓히려면 Spotify의 별도 심사(Extended Quota Mode) 필요

**스피커 노트**: 한계를 숨기지 않고 명확히 짚는 게 오히려 신뢰도를 높입니다. "왜 이런 제약이 생겼는지"는 슬라이드 3(난관)과 연결해서 설명하면 자연스럽습니다.

## 슬라이드 12. Playlist Maker 개선 방안

한계 항목과 1:1로 대응시켜 구성하면 설득력이 높습니다.

| 한계 | 개선 방안 |
|---|---|
| 태깅이 근사치 | 사용자가 곡에 직접 무드/태그를 수정·피드백할 수 있는 기능 추가 검토 |
| "비슷한 곡" 정확도 | 아티스트·장르 조합 로직을 다듬고, 필요 시 외부 음악 메타데이터 API 보강 검토 |
| 가사 검색 미지원 | Musixmatch 등 외부 가사 API 연동 검토 (Spotify API만으로는 불가능) |
| 최대 5명 제한 | **Spotify Extended Quota Mode 심사 신청**을 통한 사용자 수 제한 해제 |
| 재생 시간 채우기 단순함 | BPM/에너지 흐름을 고려한 곡 배치 로직 고도화 |
| 결과 커스터마이징 부족 | 완성된 플레이리스트 커버 이미지 직접 꾸미기 기능 추가 |
| 반복 설정의 번거로움 | 자주 쓰는 필터 조합을 저장해두고 재사용하는 히스토리 기능 추가 |

**스피커 노트**: 마무리 슬라이드 직전이므로, 이 프로젝트가 "여기서 끝"이 아니라 **실제 서비스로 키워갈 로드맵이 있다**는 인상으로 마무리하면 좋습니다.
